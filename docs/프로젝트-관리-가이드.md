# Windmill Project Hub — 프로젝트 관리 가이드

## 1. 개요

Windmill Project Hub는 **GitHub Issues + VS Code Extension + Claude Code(AI)**를 결합한 프로젝트 관리 시스템입니다.

기존 프로젝트 관리 도구(Jira, Notion 등)의 복잡성을 줄이고, 개발자가 VS Code를 떠나지 않고도 프로젝트를 계획하고 관리할 수 있도록 설계되었습니다.

### 핵심 원칙

- **GitHub Issues가 유일한 진실의 원천** — 모든 작업은 이슈로 관리
- **라벨 기반 상태/유형 관리** — 별도 프로젝트 보드 없이 라벨만으로 칸반 보드 구성
- **AI 보조 프로젝트 관리** — Claude Code가 MCP 서버를 통해 이슈를 직접 생성/분해/관리
- **VS Code 통합** — 사이드바에서 대시보드, 칸반, 이슈 상세를 바로 확인

## 2. 시스템 구성

### 2.1 인프라

```
Windmillsoft (GitHub Org)
  ├── .github/                    # Org 공통 이슈 템플릿 8종
  ├── project-template/           # 새 프로젝트 시작 템플릿 (CLAUDE.md + MCP 설정)
  ├── vscode-project-hub/         # VS Code Extension 소스
  └── [프로젝트 레포들]            # project-template 기반으로 생성
```

| 구성 요소 | 역할 |
|-----------|------|
| **VS Code Extension** | 사이드바에서 대시보드/칸반/이슈 상세/채팅/파일 관리 |
| **MCP 서버** | Claude Code가 GitHub 이슈를 직접 조회/생성/수정할 수 있는 도구 제공 |
| **Org 이슈 템플릿** | 모든 레포에서 공통 이슈 양식 자동 적용 |
| **CLAUDE.md** | Claude Code에게 프로젝트 규칙과 이슈 생성 방법을 지시 |

### 2.2 VS Code Extension 기능

| 탭 | 기능 |
|----|------|
| **Dashboard** | 이슈/PR 통계, 최근 활동, 팀 프레전스 |
| **Kanban** | `status:*` 라벨 기반 칸반 보드 (드래그 앤 드롭으로 상태 변경) |
| **Chat** | Supabase 기반 실시간 팀 채팅 (선택 사항) |
| **Files** | GitHub Release 또는 Supabase Storage 기반 파일 공유 (선택 사항) |

추가 기능:
- **My Tasks** (사이드바 트리뷰) — 현재 레포에서 나에게 할당된 이슈 목록
- **이슈 상세** — 칸반 카드 클릭 시 이슈 본문, 댓글, Sub-Issues 표시
- **프로젝트 생성** — Extension 내에서 GitHub 레포 생성 → Clone → VS Code에서 열기

## 3. 이슈 계층 구조

프로젝트의 작업을 4단계 계층으로 분해합니다. 각 계층은 GitHub의 Sub-Issues 기능으로 부모-자식 관계를 형성합니다.

```
업무영역 (Level 1)
  └── 상세기능 (Level 2)
        └── 개발항목 (Level 3)
              └── 단위작업 (Level 4)
```

| 계층 | 한국어 | 라벨 | 설명 | 예시 |
|------|--------|------|------|------|
| Level 1 | **업무영역** | `type:업무영역` | 최상위 업무 범주 | "딜러 포털 장비 관리" |
| Level 2 | **상세기능** | `type:상세기능` | 사용자 관점의 기능 단위 | "장비 등록 기능" |
| Level 3 | **개발항목** | `type:개발항목` | 구현해야 할 개발 항목 | "장비 등록 API 구현" |
| Level 4 | **단위작업** | `type:단위작업` | 최소 작업 단위 | "Equipment 모델 생성" |

### 분해 원칙

- **업무영역**은 프로젝트 시작 시 PM이 정의
- **상세기능**은 업무영역을 사용자 시나리오 단위로 분해
- **개발항목**은 상세기능을 기술적으로 분해 (프론트엔드/백엔드/DB 등)
- **단위작업**은 개발항목을 1명이 수행 가능한 최소 단위로 분해
- Claude Code에게 "업무영역 #N을 상세기능으로 분해해줘"와 같이 요청하면 AI가 분석 후 제안

## 4. 라벨 체계

### 4.1 유형 라벨 (`type:*`)

이슈의 종류를 나타냅니다. 하나의 이슈에 유형 라벨은 1개만 부여합니다.

| 라벨 | 색상 | 용도 |
|------|------|------|
| `type:업무영역` | 보라색 `#5319e7` | Level 1 이슈 |
| `type:상세기능` | 파랑색 `#1d76db` | Level 2 이슈 |
| `type:개발항목` | 초록색 `#0e8a16` | Level 3 이슈 |
| `type:단위작업` | 노랑색 `#fbca04` | Level 4 이슈 |

### 4.2 상태 라벨 (`status:*`)

이슈의 진행 상태를 나타내며, 칸반 보드의 컬럼과 1:1 매핑됩니다.

| 라벨 | 색상 | 칸반 컬럼 | 설명 |
|------|------|----------|------|
| `status:backlog` | 회색 `#808080` | Backlog | 아직 착수하지 않은 항목 |
| `status:todo` | 파랑색 `#0075ca` | To Do | 이번 스프린트에 할 일 |
| `status:in-progress` | 노랑색 `#e4e669` | In Progress | 진행 중 |
| `status:review` | 하늘색 `#a2eeef` | Review | 코드 리뷰 또는 검토 중 |
| `status:done` | 초록색 `#0e8a16` | Done | 완료 (이슈 자동 Close) |

### 4.3 기타 라벨

| 라벨 | 용도 |
|------|------|
| `enhancement` | 기능 요청 |
| `bug` | 버그 리포트 |
| `tech-review` | 기술 검토 |
| `change-request` | 변경 요청 |

> **참고**: `type:*`와 `status:*` 라벨은 Extension이 레포에 접속할 때 자동으로 생성됩니다.

## 5. 이슈 템플릿

`Windmillsoft/.github` 레포에 8종의 이슈 템플릿이 등록되어 있으며, 자체 템플릿이 없는 모든 Org 레포에 자동 적용됩니다.

| 템플릿 | 용도 | 기본 라벨 |
|--------|------|-----------|
| 업무영역 | 최상위 업무 범주 정의 | `type:업무영역`, `status:backlog` |
| 상세기능 | 사용자 관점 기능 단위 정의 | `type:상세기능`, `status:backlog` |
| 개발항목 | 구현할 개발 항목 정의 | `type:개발항목`, `status:backlog` |
| 단위작업 | 최소 단위 작업 정의 | `type:단위작업`, `status:todo` |
| 기능요청 | 새 기능 또는 개선 제안 | `enhancement` |
| 기술검토 | 기술적 의사결정/리서치 | `tech-review` |
| 버그리포트 | 버그 보고 | `bug` |
| 변경요청 | 기존 기능 변경 요청 | `change-request` |

각 템플릿은 GitHub Issue Forms(YAML 기반)으로 구성되어 있으며, 필수 필드(`required: true`)가 지정되어 있어 빈 이슈 생성을 방지합니다.

## 6. AI 보조 워크플로 (Claude Code + MCP)

### 6.1 구조

```
사용자 ←→ Claude Code ←→ MCP 서버 ←→ GitHub API
                              ↑
                          CLAUDE.md (규칙)
```

- **CLAUDE.md**: 프로젝트별 규칙 파일. Claude Code가 세션 시작 시 자동으로 읽음
- **MCP 서버**: Claude Code가 호출할 수 있는 GitHub 이슈 관리 도구 8종 제공
- **이슈 템플릿 참조**: Claude Code는 이슈 생성 전 `.github` 레포의 템플릿을 읽어 필드 구조를 확인

### 6.2 사용 가능한 MCP 도구

| 도구 | 설명 |
|------|------|
| `get_issue` | 이슈 상세 조회 (본문, 라벨, 댓글) |
| `list_issues` | 이슈 목록 조회 (필터 지원) |
| `list_labels` | 레포 라벨 목록 |
| `get_hierarchy` | Sub-Issues 트리 조회 |
| `create_issue` | 새 이슈 생성 |
| `create_sub_issue` | Sub-Issue 생성 + 부모 연결 |
| `update_issue` | 이슈 수정 |
| `assign_issue` | 담당자 배정 |

### 6.3 주요 AI 워크플로

#### 업무 분해

```
사용자: "업무영역 #5를 상세기능으로 분해해줘"

Claude:
1. get_issue(5) → 이슈 내용 확인
2. get_hierarchy(5) → 기존 하위 구조 확인
3. 분석 후 상세기능 목록 제안
4. (사용자 승인 후)
5. create_sub_issue(5, "상세기능: 장비 등록", labels=["type:상세기능", "status:backlog"]) × N
```

#### 진행 상태 변경

```
사용자: "#12 이슈를 진행 중으로 바꿔줘"

Claude:
1. update_issue(12, remove_labels=["status:todo"], add_labels=["status:in-progress"])
```

#### 담당자 배정

```
사용자: "#12~#15 이슈를 홍길동에게 배정해줘"

Claude:
1. assign_issue(12, assignees=["honggildong"])
2. assign_issue(13, assignees=["honggildong"])
3. ...
```

#### 현황 파악

```
사용자: "현재 진행 중인 이슈 목록 보여줘"

Claude:
1. list_issues(labels=["status:in-progress"])
2. 결과를 정리하여 표시
```

## 7. 새 프로젝트 시작하기

### 방법 1: Extension에서 생성

1. VS Code에서 Windmill Project Hub Extension 열기
2. 레포 선택 화면에서 **"+ Create New Project"** 클릭
3. 이름, 설명, 공개/비공개, 조직 선택 후 **"Create & Clone"**
4. VS Code가 자동으로 Clone → 폴더 열기
5. Extension이 git remote를 감지하여 자동 초기화

### 방법 2: GitHub에서 템플릿으로 생성

1. GitHub에서 `Windmillsoft/project-template` 레포 → **"Use this template"**
2. 새 레포 이름 입력 후 생성
3. 로컬에 Clone
4. VS Code에서 열면 Extension이 자동 초기화

### 생성 후 자동으로 적용되는 것들

| 항목 | 출처 | 자동 여부 |
|------|------|----------|
| 이슈 템플릿 8종 | `Windmillsoft/.github` | 자동 (Org fallback) |
| `status:*` 라벨 5종 | Extension `ensureKanbanLabels` | 자동 (레포 접속 시) |
| `type:*` 라벨 4종 | Extension `ensureKanbanLabels` | 자동 (레포 접속 시) |
| CLAUDE.md | `project-template` | 자동 (템플릿 사용 시) |
| MCP 설정 | `project-template` | 자동 (템플릿 사용 시) |

## 8. 일상적인 사용 흐름

### 8.1 프로젝트 계획 단계

```
1. PM이 업무영역 이슈 생성 (GitHub 웹 또는 Claude Code)
2. Claude Code에게 업무영역 → 상세기능 분해 요청
3. 상세기능 검토 후 승인
4. Claude Code에게 상세기능 → 개발항목 분해 요청
5. 필요시 개발항목 → 단위작업 추가 분해
6. 담당자 배정
```

### 8.2 개발 단계

```
1. VS Code Extension에서 My Tasks 확인
2. 작업할 이슈의 상태를 status:in-progress로 변경
3. 개발 진행
4. 완료 후 status:review로 변경
5. 코드 리뷰 후 status:done으로 변경 (이슈 자동 Close)
```

### 8.3 모니터링

```
1. Extension Dashboard에서 전체 현황 확인
2. Kanban 보드에서 진행 상태 한눈에 파악
3. Claude Code에게 "현재 진행 상황 요약해줘" 요청
```
